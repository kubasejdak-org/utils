# Contributing Guidelines

First off, thanks for taking the time to contribute.

This project is open source and welcomes contributions, discussions, and differing perspectives. That said, it is also
intentionally opinionated. Please read the following guidelines carefully before investing significant time in a
contribution.

## Philosophy

This repository reflects a specific set of design decisions, trade-offs, and opinions. Contributions are evaluated not
only on technical merit but also on alignment with the overall direction of the project.

Constructive disagreement is welcome. However, alignment with the project's goals is required for changes to be
accepted.

## Ground Rules

- Be respectful and professional in all interactions.
- Focus discussions on technical and conceptual merit.
- Avoid bikeshedding and low-value debates.
- Assume good intent, but be precise in your arguments.

## Before You Start

- Check existing issues and pull requests to avoid duplication.
- If your change is non-trivial, open an issue first to discuss it.
- Make sure your proposal aligns with the project’s direction.

## Contribution Types

### Bug Fixes

- Clearly describe the issue.
- Provide reproducible steps if possible.
- Keep fixes minimal and focused.

### Features

- Discuss first via an issue.
- Explain the motivation and use case.
- Be prepared that not all feature requests will be accepted.

### Documentation

- Improvements are always welcome.
- Keep documentation concise and aligned with the project's tone.

## Pull Request Process

1. Fork the repository.
2. Create a focused branch (`feature/...`, `fix/...`).
3. Keep changes small and self-contained.
4. Write clear commit messages.
5. Open a pull request with a detailed description.

## Review & Decision Process

All contributions are reviewed, but not all will be accepted.

Key points:

- Feedback may be direct and to the point.
- Iteration may be required before acceptance.
- Some proposals may be declined even if technically valid.

### Final Decision

This project is curated and opinionated.

While open discussion is encouraged, **the final decision on whether a change is accepted — and in what form — rests
with the maintainer**.

This includes:

- Accepting or rejecting contributions
- Modifying proposed solutions
- Defining the long-term direction of the project

If this model does not align with your expectations, this project may not be a good fit for your contributions.

## Code Style & Quality

- Follow existing patterns in the codebase.
- Prefer clarity over cleverness.
- Avoid unnecessary abstractions.
- Keep dependencies minimal.

## Communication

- Use GitHub Issues for discussions.
- Keep conversations focused and actionable.
- Avoid long philosophical debates unless they directly impact the project.

## License

By contributing, you agree that your contributions will be licensed under the same license as this project.

## Closing Notes

Contributions are appreciated, but they are not automatically accepted.

The goal is to maintain a coherent, high-quality, and consistent codebase — not to accommodate every possible idea.

If you’re aligned with that, you’re welcome here.
